Vue3 为什么删除了 $on/$off/$once

sleepkele2026/4/15大约 3 分钟

Vue3 为什么删除了 $on/$off/$once

一、Vue3 为什么删除了 $on / $off / $once？
✅ 结论

Vue3 删除实例事件 API，是为了 消除隐式 EventBus 模式，让组件通信更明确、更可追踪、更利于 Tree-shaking 和类型推导。

🔎 Vue2 的问题

在 Vue2 里，每个组件实例都是一个小型 EventEmitter：

this.$on('xxx')
this.$emit('xxx')

问题有三个：

❌ 1. 容易被滥用成全局 EventBus

很多人写：

export const bus = new Vue()
bus.$on(...)

问题：

  • 隐式依赖
  • 难排查
  • 生命周期难管理
  • 容易内存泄漏

Vue3 官方直接说：不推荐这种模式。

❌ 2. 破坏单向数据流

Vue 推崇：

props down
events up

但 $on 允许任意监听，容易变成：谁都能监听谁

架构变混乱。

❌ 3. Tree-shaking 不友好

Vue3 是基于 ES Module 的。

如果实例上默认带：

$on
$off
$once

就意味着：

👉 这套事件系统永远打包进核心。

而 Vue3 的目标是：

让无用 API 可以被打包器删除。

🚀 Vue3 的替代方案

Vue3 保留了：

$emit

删除了：

$on / $off / $once

组件事件现在只能：

👉 父组件监听
👉 子组件 $emit

如果需要 EventBus：

import mitt from 'mitt'

官方推荐用外部库。

二、组件事件 vs DOM 事件底层区别

1️⃣ DOM 事件

<button @click="fn">

底层机制：

在 patch 阶段处理：

源码位置（Vue2）：

platforms/web/runtime/modules/events.js

核心逻辑：

el.addEventListener(event, handler)

本质：

浏览器原生事件系统

2️⃣ 组件事件

this.$emit('my-event')

底层机制：

vm._events = {
  'my-event': [fn1, fn2]
}

触发：

const cbs = vm._events[event]
for (...) cbs[i]()

本质：

发布订阅模式（纯 JS 实现）

🔥 关键区别

DOM 事件组件事件
存储位置DOM 节点vm._events
执行机制浏览器JS 数组遍历
传播机制冒泡/捕获不冒泡
生命周期管理patch 阶段initEvents

三、_hasHookEvent 优化点在哪？

这个是性能优化。

场景

Vue2 允许监听生命周期：

this.$on('hook:mounted', fn)

如果没有优化：

每次执行 mounted 都要：

if (vm._events['hook:mounted']) ...

这是额外开销。

Vue 做法

在 $on 里：

if (hookRE.test(event)) {
  vm._hasHookEvent = true
}

在执行生命周期时：

if (vm._hasHookEvent) {
  vm.$emit('hook:mounted')
}

🔥 优化点：

如果从来没人监听 hook 事件，就完全跳过 emit 逻辑。

这是典型的：

flag short-circuit optimization

四、updateListeners 是怎么 diff 的？

这个是事件系统的核心算法。

它做什么？

当父组件重新渲染时：

<child @a="fn1"/>

变成：

<child @a="fn2"/>

Vue 要：

  • 移除旧监听
  • 添加新监听

核心源码逻辑（简化版）

function updateListeners(on, oldOn, add, remove) {
  for (name in on) {
    const cur = on[name]
    const old = oldOn[name]

    if (!old) {
      add(name, cur)
    } else if (cur !== old) {
      old.fns = cur
    }
  }

  for (name in oldOn) {
    if (!on[name]) {
      remove(name, oldOn[name])
    }
  }
}

🔎 分析三种情况
1️⃣ 新增事件

old: {}
new: { click: fn }

→ add

2️⃣ 修改事件

old: { click: fn1 }
new: { click: fn2 }

不会 remove + add
而是：

oldInvoker.fns = newFn

🔥 关键优化：

不重新注册 DOM 事件
只替换内部函数引用

3️⃣ 删除事件

old: { click: fn }
new: {}

→ remove

为什么要用 invoker？

Vue 不直接绑定 fn，而是包装：

function invoker(e) {
  invoker.fns(e)
}

好处：

  • DOM 上的 handler 不变
  • 只替换 fns
  • 性能更高

🎯 总结

Vue2 的组件事件是基于发布订阅模式实现的，每个组件实例维护一个 _events 容器；DOM 事件则是通过 patch 阶段调用原生 addEventListener。Vue3 删除 $on/$off/$once 是为了避免 EventBus 滥用、强化单向数据流并支持 Tree-shaking。_hasHookEvent 是为了避免生命周期触发时的额外事件判断。updateListeners 通过对比新旧 listeners，实现新增、更新和删除的精细 diff，并通过 invoker 机制避免重复绑定 DOM 事件，从而提升性能。`
