一、加载性能（Loading）

1. HTTP请求优化

HTTP请求优化是前端性能优化的核心环节，直接影响页面加载速度和用户体验。以下从协议升级、请求数量、请求大小、缓存策略、加载顺序五个维度进行深度剖析，并给出具体解决方案：

1.1 协议升级：从 HTTP/1.1 到 HTTP/3

核心问题：HTTP/1.1的队头阻塞（Head-of-line Blocking）、串行请求限制
解决方案：

  1. 升级到HTTP/2

    • 二进制分帧：并行处理多个请求/响应
    • 多路复用：单个TCP连接处理所有请求
    • 头部压缩（HPACK）：减少HTTP头大小
    • 服务器推送：提前推送资源（如CSS、JS）

  2. 进一步升级到HTTP/3

    • 基于QUIC协议：解决TCP层的队头阻塞
    • 更快的连接建立：0-RTT或1-RTT握手
    • 更好的移动网络支持：抗丢包能力强
# nginx.conf
# Nginx 配置 HTTP/2
server {
    listen 443 ssl http2;  # 启用HTTP/2
    server_name example.com;
    
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
    
    # 其他HTTPS配置...
}

实施建议：
大多数CDN（如Cloudflare、AWS CloudFront）已支持HTTP/3，可通过HTTPS强制跳转实现升级。

在 Cloudflare 控制面板：
进入域名→网络→启用 HTTP/3（QUIC）
启用 "自动 HTTPS 重定向"

1.2 减少请求数量

核心问题：过多HTTP请求导致的TCP握手和慢启动
解决方案：

  1. 资源合并与内联

    • CSS Sprites：合并小图标为一张大图
    • 内联关键CSS：减少渲染阻塞请求
    • JS/CSS文件合并：避免零碎请求（需权衡缓存失效问题）
    # 使用spritesmith生成雪碧图
npm install spritesmith --save-dev

# sprite.js
const Spritesmith = require('spritesmith');
Spritesmith.run({
  src: ['icons/*.png'],
  dest: 'dist/sprite.png',
  algorithm: 'top-down'
}, function handleResult(err, result) {
  // 生成对应的CSS
});

  2. 代码分割与懒加载

    • 使用动态导入（Dynamic Import）拆分JS包
    • 路由懒加载（如React.lazy、Vue async components）
    • 图片懒加载（Intersection Observer API）
    <!-- HTML 图片懒加载 -->
<img data-src="image.jpg" class="lazy-load" alt="风景图">

<!-- JavaScript -->
document.addEventListener("DOMContentLoaded", () => {
  const lazyImages = document.querySelectorAll('img.lazy-load');
  
  if ('IntersectionObserver' in window) {
    const imageObserver = new IntersectionObserver((entries, observer) => {
      entries.forEach(entry => {
        if (entry.isIntersecting) {
          const img = entry.target;
          img.src = img.dataset.src;
          img.classList.remove('lazy-load');
          imageObserver.unobserve(img);
        }
      });
    });
    
    lazyImages.forEach(img => imageObserver.observe(img));
  }
});

  3. HTTP/2服务器推送

    • 预推送关键资源（如字体、首屏CSS）
    # Nginx配置示例
http2_push /styles/main.css;
http2_push /scripts/main.js;
http2_push /fonts/inter.woff2;

  4. 数据聚合API

    • 将多个API请求合并为一个（如GraphQL）
    • 使用服务端中间层聚合数据（如BFF模式）

1.3 减小请求大小

核心问题：大文件导致的长时间传输
解决方案：

  1. 压缩与转码

    • Gzip/Brotli压缩：文本资源压缩率可达70-90%
    # Nginx启用Brotli
brotli on;
brotli_types text/plain text/css application/json;
brotli_types text/plain text/css application/json application/javascript;
    • 图片格式转换：JPEG→WebP/AVIF（体积减小30-50%）
    const sharp = require('sharp');

 async function convertToWebp(inputPath, outputPath) {
   await sharp(inputPath)
     .webp({ quality: 80 })
     .toFile(outputPath);
 }
    • 视频转码：H.264→AV1（同等画质下体积减半）

  2. 移除冗余代码

    • Tree Shaking：移除未使用的JS/CSS
    • 代码压缩：UglifyJS、Terser压缩JS
    • CSS Purge：移除未使用的CSS（如Tailwind CSS的purge功能）
    // tailwind.config.js Tailwind CSS 移除未使用样式
 module.exports = {
   purge: {
     enabled: process.env.NODE_ENV === 'production',
     content: [
       './src/**/*.html',
       './src/**/*.js',
     ],
   },
   // 其他配置...
 }

  3. 字体优化

    • 子集化（Subsetting）：只包含页面使用的字符
    • WOFF2格式：比TTF/OTF小30%
    • 字体预加载（Preload）：
    <link rel="preload" href="/fonts/Inter.woff2" as="font" crossorigin>

1.4、缓存策略优化

核心问题：重复请求相同资源
解决方案：

  1. 强缓存（Cache-Control/Expires）

    • 静态资源设置长缓存（1年+）：
    Cache-Control: max-age=31536000, immutable
     // 设置强缓存（Express.js）
 const express = require('express');
 const app = express();

 // 静态资源设置1年缓存
 app.use('/static', express.static('public', {
   maxAge: 31536000000  // 1年
 }));
    • 文件名哈希：更新资源时变更文件名（如main.abc123.js）

  2. 协商缓存（ETag/Last-Modified）

    • 动态资源使用ETag：
    ETag: "1234567890"

  3. Service Workers缓存

    • 离线缓存：拦截网络请求，使用本地副本
    • 预缓存关键资源：
    self.addEventListener('install', event => {
  event.waitUntil(
    caches.open('v1').then(cache => {
      return cache.addAll(['/', '/index.html', '/main.css']);
    })
  );
});

  4. CDN缓存

    • 设置CDN缓存规则：
    Cache-Control: public, max-age=86400
    • 使用CDN预热工具：批量刷新缓存

1.5、优化请求顺序

核心问题：关键资源加载延迟
解决方案：

  1. 资源优先级控制

    • 预加载（Preload）关键资源：
    <link rel="preload" href="/critical.js" as="script">
    • 预连接（preconnect）重要域名：
    <link rel="preconnect" href="https://api.example.com">

  2. 懒加载非关键资源

    • 非首屏内容延迟加载：
    // 使用Intersection Observer懒加载图片
const observer = new IntersectionObserver(entries => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      observer.unobserve(img);
    }
  });
});

  3. HTTP/2优先级调整

    • 通过服务器配置设置资源权重：
    # Nginx设置HTTP/2优先级
http2_push_priority /critical.css weight=1;
http2_push_priority /non-critical.js weight=5;

1.6、监控与工具链

关键指标：

  • 首字节时间（TTFB）
  • 资源加载时间（Performance API）
  • 缓存命中率（Chrome DevTools Network面板）

实用工具：

  1. Chrome DevTools：分析请求瀑布图
  2. Lighthouse：评估HTTP优化效果
  3. WebPageTest：多地点网络测试
  4. HTTP Archive：行业性能基准对比

1.7、高级优化策略

  1. HTTP/2服务器推送策略

    • 仅推送关键路径资源，避免过度推送
    • 使用HTTP/2 Push Promise头：
    Link: </style.css>; rel=preload; as=style

  2. QUIC协议优化

    • 启用0-RTT连接恢复
    • 配置适当的拥塞控制算法（如BBR）

  3. 边缘计算

    • 在CDN边缘节点处理请求（如Cloudflare Workers）
    • 边缘缓存动态内容

1.8、常见误区与避坑指南

  1. 过度合并文件：导致缓存失效范围扩大
  2. 滥用服务器推送：推送未使用的资源反而降低性能
  3. 忽略移动端优化：移动网络更易受丢包影响（HTTP/3优势）
  4. 缓存策略错误：静态资源未设置长缓存，动态资源未使用协商缓存

通过系统化的HTTP请求优化，通常可实现：

  • 首屏加载时间减少30-50%
  • 带宽使用降低50-70%
  • 缓存命中率提升至90%以上

建议结合性能监控工具（如Google Analytics、New Relic）持续跟踪优化效果，并建立自动化测试流程确保优化成果。
